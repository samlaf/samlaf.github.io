## Best first tutorial: LWN, “Using the KVM API”

Start with **LWN’s “Using the KVM API”**. It walks through setting up a VM directly without QEMU, runs a tiny 16-bit x86 guest, prints via an emulated serial port, and exits via `hlt`. That maps almost perfectly to your first two goals: tiny guest blob, IO port write, and HLT handling. ([LWN.net][1])

The core structure you want to internalize is:

```c
open("/dev/kvm")
ioctl(KVM_GET_API_VERSION)
ioctl(KVM_CREATE_VM)
mmap guest RAM
ioctl(KVM_SET_USER_MEMORY_REGION)
ioctl(KVM_CREATE_VCPU)
mmap struct kvm_run
set sregs/regs
copy guest code into RAM
while (true) {
    ioctl(KVM_RUN)
    switch (run->exit_reason) {
    case KVM_EXIT_HLT:
    case KVM_EXIT_IO:
    case KVM_EXIT_MMIO:
    }
}
```

That is the “TweetNaCl-ish” essence of a KVM VMM.

## Best tiny codebase: `dpw/kvm-hello-world`

After the article, read **`dpw/kvm-hello-world`**. It is probably the cleanest minimal artifact for your purpose. It demonstrates the KVM API with a host program plus small guest payloads, including real mode, protected mode, paged mode, and long mode. The workflow is exactly the VMM skeleton: open `/dev/kvm`, create VM, map memory, create vCPU, set registers, run, handle `KVM_EXIT_HLT`, and inspect results. ([DeepWiki][2])

I would use this as the reference implementation while writing your own version from scratch.

## Best “few lines” sanity check: zserge’s KVM article

**zserge’s “KVM host in a few lines of code”** is good as a minimal mental model. It is less of a full learning path, but great for proving to yourself that a userspace process plus `/dev/kvm` really is enough to run guest code. ([zserge.com][3])

## Best next step toward Linux boot: iovec “Building a hypervisor”

For your later milestones — long mode, paging, Linux boot protocol, kernel loading, initramfs — the **iovec “Building a hypervisor”** series looks closest to what you want. The first post explicitly says the series builds a KVM-based hypervisor from scratch and eventually boots a Linux kernel. ([iovec.net][4])

That is the bridge from:

```text
toy blob + HLT + IO
```

to:

```text
load bzImage
set up boot params
enter 64-bit Linux
provide initramfs
emulate enough devices
```

## Use the kernel KVM docs as your spec

Keep the official **Linux KVM API documentation** open while coding. The important conceptual point from the docs is that KVM is an ioctl API over several file descriptor types: a system fd from `/dev/kvm`, a VM fd, vCPU fds, and sometimes device fds. ([Kernel.org][5])

That distinction matters because your VMM will naturally organize around those objects:

```text
Kvm
  Vm
    MemoryRegion
    Vcpu
    DeviceModel
```

## Where virtio should enter

Do **not** start with virtio. Add it after you can boot Linux with a serial console/initramfs.

For virtio, first read the Linux kernel virtio overview. Virtio is a paravirtual device protocol where the guest driver and host device model communicate through shared data structures, typically virtqueues. ([Kernel Docs][6])

For the VMM learning path, I would add devices in this order:

```text
1. HLT exit
2. I/O port debug console, e.g. port 0xE9 or COM1-ish output
3. Simple MMIO toy device
4. Linux boot with no disk, just initramfs
5. virtio-console over MMIO
6. virtio-blk over MMIO
```

`virtio-blk` is a bigger jump than it looks because you need queues, descriptors, available/used rings, interrupts, feature negotiation, and a backend block file. The old Linux-KVM virtio block page is useful context, but for implementing a toy backend, the kernel virtio docs and a tiny VMM codebase will be more useful. ([Linux KVM][7])

## My recommended sequence

Follow this order:

1. **Read LWN “Using the KVM API.”** Implement the tiny 16-bit guest yourself, not by copy-paste. ([LWN.net][1])
2. **Read `dpw/kvm-hello-world`.** Compare your version to its real/protected/paged/long-mode structure. ([DeepWiki][2])
3. **Add `KVM_EXIT_IO` debug output.** Use port `0xE9` or `0x3f8`.
4. **Add `KVM_EXIT_MMIO`.** Make a fake memory-mapped device like “write byte to `0xf0000000` prints char.”
5. **Follow iovec for Linux boot.** This is where you learn long mode, Linux boot protocol, boot params, and initramfs. ([iovec.net][4])
6. **Only then implement virtio-mmio.** Start with console, then block.
